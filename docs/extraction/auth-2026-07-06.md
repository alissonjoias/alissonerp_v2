## 4. Auth Flow (End-to-End)

### 4.1 Login Flow

```
User enters credentials 
  → src/app/(auth)/login/page.tsx (Line 38):
       supabase.auth.signInWithPassword({ email, password })
         → Supabase GoTrue validates
         → Returns JWT session (stored in cookies: sb-access-token, sb-refresh-token)
  → router.push(next) → proxy.ts middleware fires
```

### 4.2 Middleware (proxy.ts) Auth Check

**Source:** `src/proxy.ts` (Lines 139–269)
**Matcher:** All routes EXCEPT `_next/static`, `_next/image`, `favicon.ico`, and static assets (Line 266–268)

**Whitelist — routes that BYPASS ALL auth checks** (Lines 145–188):
- `"/"` (root page)
- `"/api/*"` (ALL API routes — auth checked per-route!)
- `"/login"` (login page)
- `"/orcamento"` (public budget/orcamento page)
- `"/_next/*"` (Next.js assets)
- `"/favicon*"`, `"/logo*"` (static assets)
- Any path containing `"."` (static files)

**Protected page routes** (Lines 190–260):
1. Non-whitelisted pages: check `supabase.auth.getUser()` with 30s session cache
2. `!user` → redirect to `/login?next=<original_path>`
3. If target is `/service` or `/service-painel` → auth check only (role checked by page component)
4. If prefix matches `PREFIXOS_PROTEGIDOS` → call `/api/user/permissoes?user_id=<UUID>` to get modules, check if module allows access. If `!modulos || !canAccessModulos` → redirect to `/`

**IMPORTANT:** The middleware does NOT check auth for `/api/*` routes — each API route is responsible for its own auth. This creates a risk: if a developer forgets to add `authOr401()` to a new API route, it will be publicly accessible (CONFLITO-002).

**Rate Limiting** (Lines 19–44, 157–176):
- 5000 req/min for unauthenticated API calls
- 10000 req/min for authenticated API calls
- Enforced per IP in-memory (not persistent)

### 4.3 Server-Side Auth (requireAuth)

**Source:** `src/lib/auth-server.ts` (Lines 1–81)

```
route handler → requireAuth(req) or authOr401(req)
  → createServerSupabase() → supabase.auth.getUser() [cached, TTL 30s]
  → !user? → throw AuthError (401)
  → SELECT role, ativo FROM alissonerp.usuarios WHERE id = <UUID>
  → !row? → throw AuthError "Usuário não encontrado"
  → row.ativo === false? → throw AuthError "Usuário desativado..."
  → return { id, role, email }
```

**BLINDAGEM:** The `ativo` flag check was added because deactivated users (ex-employees) still had valid Supabase sessions and could operate the ERP normally.

### 4.4 Session Caching

**Server-side cache** (`src/lib/supabase/server.ts` Lines 8–66):
- `userCache`: Map<cookieString, { user, exp }>, TTL = 30,000ms (30s)
- Cleaner interval: every 60s
- Max 1000 entries; evicts oldest on overflow
- Bypassed if `jwt` parameter is explicitly passed

**Session cache** (`src/proxy.ts` Lines 47–56, 214–230):
- `sessionCache`: Map<cookieHeader, { user, exp }>, TTL = 30,000ms (30s)
- Cleaner interval: every 60s
- Max 1000 entries

**Permission cache** (`src/proxy.ts` Lines 89–130):
- `rolePermCache`: Map<userId, { modulos, exp }>, TTL = 15 * 60 * 1000ms (15 min)
- Max 500 entries; evicts oldest on overflow
- Fetches from `/api/user/permissoes` via internal `http://127.0.0.1:${port}`

### 4.5 Client-Side Auth (useAuth hook)

**Source:** `src/hooks/useAuth.ts` (Lines 1–153)

```
useAuth() hook
  → Check cache (module-global, TTL 5 min)
  → If cache valid → return cached
  → If inflight promise → wait for it (deduplication)
  → Otherwise:
    1. supabase.auth.getUser()
    2. !user? → return null values
    3. Parallel fetch:
         GET /api/user/role?user_id=<UUID>
         GET /api/user/permissoes?user_id=<UUID>
    4. If either returns 403 (deactivated user) → signOut + redirect to /login?reason=desativado
    5. Merge: role from permissoes endpoint, nome from role endpoint, avatar from role endpoint
    6. Return: { user, role, nome, fotoUrl, loading, modulos_leitura, modulos_escrita, isOperadorService }
```

**Invalidation:** `invalidarCacheAuth()` (exported) — clears all caches. Called after role/permission mutations.

**OAuth2 Keepalive** (Lines 139–150): admin/dev users get a 30-min interval calling `/api/admin/tiny-keepalive` to refresh Tiny ERP token.

### 4.6 Permission Check (temPermissao)

**Source:** `src/lib/permissoes.ts` (Lines 1–44)

```
temPermissao(userId, acao, role)
```

## 5. defineRoute() Pattern

**Source:** `src/lib/api-handler.ts` (Lines 1–92)

**Signature:**
```typescript
defineRoute<TInput>({
  requireRoles?: string[],      // inline role gate
  inputSchema?: ZodType<TInput>,  // Zod validation (auto-registers in OpenAPI)
  meta?: RouteMeta,              // OpenAPI metadata (path, method, tags, summary)
  handler: (req, ctx) => Promise<NextResponse>
})
```

**Auth flow within `defineRoute`:**
1. `requireAuth(req)` — gets session + role from DB (checks `ativo` flag)
2. `requireRoles` check: if user.role is NOT in the array → ForbiddenError (403)
3. `inputSchema` check: parses JSON body, returns 400 with Zod issues on failure
4. Passes `{ user: AuthUser, body: TInput, params? }` to handler
5. Catches AuthError → 401, ForbiddenError → 403, everything else → 500

**Key observation:** `defineRoute` is NOT widely adopted. Only 1 route found uses it (from grep results). Most routes use the classic pattern:
```typescript
const [user, errRes] = await authOr401(req);
if (errRes) return errRes;
// ... manual role checks, manual Zod parsing
```
This means `authOr401()` is the dominant pattern — used in 100+ routes.

**Critical problems with current RLS:**
1. Most tables grant `'admin'` full access but NOT `'desenvolvedor'` — the `fn_user_role()` function is the check, and it queries `alissonerp.usuarios` for the current user's role. This means RLS policies treat admin and desenvolvedor DIFFERENTLY at the database level, even though the application treats them identically (CONFLITO-003).
2. Migration 070's `notificacoes_admin` policy correctly uses `IN ('admin','desenvolvedor')`, but migration 068's policies only check for `'admin'`.
3. `gerente_fabrica` has full RLS access to several production tables (ordens, pcp, distribuicoes, movimentacoes_estoque, protocolos) but NO other roles (gerente_loja, ourives besides SELECT, etc.) have write access at the DB level — meaning the application layer (API routes) must be the primary authorization mechanism.

## 10. `fn_user_role()` — The RLS Helper Function

**Source:** `supabase/migrations/068_role_varchar.sql` (Lines 33–36)

```sql
CREATE OR REPLACE FUNCTION alissonerp.fn_user_role()
RETURNS VARCHAR(50) AS $$
  SELECT role FROM alissonerp.usuarios WHERE id = auth.uid();
$$ LANGUAGE sql SECURITY DEFINER STABLE;
```

**Characteristics:**
- `SECURITY DEFINER` — runs with privileges of the function owner (typically postgres/superuser)
- `STABLE` — returns same result for same arguments within a transaction
- Used in ALL RLS policies to determine the current user's role
- Returns `NULL` if user not found in `usuarios` table
- This function is the SINGLE POINT OF TRUTH for RLS role resolution

## 11. Auth Trigger (fn_handle_new_user)

**Source:** `supabase/migrations/017_auth_trigger.sql` (Lines 5–25), `068_role_varchar.sql` (Lines 39–62)

```sql
AFTER INSERT ON auth.users
  → INSERT INTO alissonerp.usuarios (id, email, nome, role)
    VALUES (
      NEW.id,
      NEW.email,
      COALESCE(
        NEW.raw_user_meta_data->>'nome',
        NEW.raw_user_meta_data->>'full_name',
        split_part(NEW.email, '@', 1)  -- fallback: email prefix
      ),
      COALESCE(NEW.raw_user_meta_data->>'role', 'ourives')  -- fallback: ourives
    )
    ON CONFLICT (id) DO NOTHING
```

**Key behaviors:**
- User creation trigger is on `auth.users` (Supabase-managed)
- Default role: `'ourives'` (most restricted)
- `raw_user_meta_data` carries `nome` and `role` from the creation API
- `ON CONFLICT DO NOTHING` — idempotent, does not overwrite existing profiles

## 15. Permission Sharing: `admin` vs `desenvolvedor`

Throughout the codebase, `"admin"` and `"desenvolvedor"` are treated as equivalent in ALMOST every check:
- `isFullAccess()` (roles.ts:18–20) — both pass
- `canAccess()` (permissions.ts:218–223) — both pass with explicit string check
- `ROLE_MENUS` (permissions.ts:81–103) — identical menu arrays for both roles
- `defineRoute()` `requireRoles` — checked via array include
- Sidebar injection arrays — both typically included

**BUT at the RLS level** (see Section 9.3), most policies only check for `'admin'`, NOT `'desenvolvedor'`. Only `notificacoes_admin` (migration 070) and `comentarios_delete` (migration 070) include both. This means `desenvolvedor` users may have LESS database-level access than `admin` users for direct Supabase queries, despite having identical application-level permissions (CONFLITO-003).

## 17. Identified Issues

### CONFLITO-001: `diretor` role — exists in roles.ts but not in permissions.ts
- **Where:** `src/lib/roles.ts` Line 2: `ROLES_FULL_ACCESS = ["admin", "desenvolvedor", "diretor"]`
- **Where missing:** `src/lib/auth/permissions.ts` Line 1–12: `Role` type does NOT include `"diretor"`
- **Impact:** `diretor` users get `isFullAccess() = true` (server-side bypass) but have NO menu in the sidebar because `getMenuGroupsForRole("diretor")` returns empty array. They can access everything via API but see a blank sidebar.
- **Severity:** HIGH — users exist in this role but may have broken UI

### CONFLITO-002: `/api/*` routes are NOT auth-checked by middleware
- **Where:** `src/proxy.ts` Lines 145–188: all `/api/*` routes are whitelisted
- **Risk:** Any new API route without `authOr401()` or `requireAuth()` is publicly accessible
- **Examples of routes that WERE missing auth:** `src/app/api/vendas/por-cliente/[clienteId]/route.ts` Line 10: "Onda 1D: rota estava SEM AUTH NENHUMA. Adicionado authOr401"
- **Severity:** MEDIUM — documented pattern, but easy to forget

### CONFLITO-003: RLS policies don't grant `desenvolvedor` same access as `admin`
- **Where:** Most policies in migration 068: `USING (fn_user_role() = 'admin')` instead of `IN ('admin','desenvolvedor')`
- **Impact:** If direct Supabase queries are used (React Query with supabase.from()), `desenvolvedor` users may get empty results where `admin` users see data
- **Exception:** `notificacoes_admin` policy (migration 070) and `comentarios_delete` (migration 070) DO include `desenvolvedor`
- **Severity:** MEDIUM — most data access goes through API routes, not direct queries

### CONFLITO-004: `fn_user_role()` returns NULL for users with no `usuarios` row
- **Where:** `fn_user_role()` definition (migration 068, Line 33-36)
- **Impact:** A user with valid `auth.users` session but missing `alissonerp.usuarios` row would get NULL from all RLS policies → effectively blocked from ALL RLS-protected tables
- **But:** `requireAuth()` in `auth-server.ts` throws AuthError if `!row`, so API routes would 401 before reaching DB queries
- **Severity:** LOW — double-protection by API layer

### SUS-001: `diretor` role is inconsistently applied
- `ROLES_FULL_ACCESS` includes `diretor` → admin bypass on server
- But NOT in `Role` type union
- Not in `ROLE_MENUS` → no sidebar
- Not seeded in `alissonerp.roles` table (migration 068 only seeds 10 roles, diretor is not among them)
- Not in `role_permissoes` seed data
- `canAccess()` explicitly checks only `"admin" || "desenvolvedor"`, ignoring `diretor`
- **Severity:** HIGH — inconsistent behavior depending on which code path is hit

### SUS-002: `gerente_loja` has NO RLS write access to any table
- All production tables' RLS policies grant write access only to `'admin'` and sometimes `'gerente_fabrica'`
- `gerente_loja` relies entirely on API route authorization (which works because API writes go through `sql` tagged templates that bypass RLS via service role or direct postgres connection)
- **Severity:** LOW — consistent with current architecture (API is the gateway)

### SUS-003: `permissoes_usuario` table only has 4 actions
- Table is designed for general granular permissions but currently only supports `abrir_purificacao`, `concluir_purificacao`, `abrir_ligacao`, `concluir_ligacao`
- The 4 seeds only apply to users with `role IN ('admin','desenvolvedor','gerente_fabrica')`
- Other modules don't use this table — they use `role_permissoes` or inline role arrays
- **Severity:** INFO — design is extensible but underutilized

### SUS-004: Role `entregador` is defined in `permissions.ts` but nowhere in `roles.ts`
- `permissions.ts` Line 90: `entregador: ["/entregas-venda"]`
- `roles.ts` has NO constant referencing `entregador`
- Not in the `Role` type (wait — it is, at Line 8)
- Actually, `entregador` IS in the `Role` type at Line 8 of permissions.ts but has ZERO exposure in `roles.ts` constants
- **Severity:** LOW — role works as defined but has no module-level constant arrays

## 21. Key Architectural Decisions

1. **API-layer authorization, not RLS-layer:** Most authorization happens in API route handlers via `authOr401()` + inline role checks. RLS is a secondary defense. This means the database's `sql` tagged template (which likely uses service_role credentials) can bypass RLS.

2. **Double source of truth for menus:** The sidebar loads modules from the database (`role_permissoes`) but falls back to hardcoded arrays (`ROLE_MENUS`). Additionally, hardcoded role arrays in `Sidebar.tsx` inject modules that the DB might not have. This creates 3 potential sources of truth.

3. **`ativo` flag as pseudo-soft-delete:** Rather than revoking Supabase sessions, deactivated users are blocked at the API layer (401/403) and forced to log out on the client.

4. **No middleware auth for APIs:** Every API route is responsible for its own authentication. This is a deliberate choice but creates a footgun for new development.

5. **Legacy user_profiles table:** `public.user_profiles` is a remnant from Directus with its own role enum (admin/gerente/consultora/financeiro/logistica/producao/user). It is kept in sync by user edit operations but is otherwise a secondary citizen.
